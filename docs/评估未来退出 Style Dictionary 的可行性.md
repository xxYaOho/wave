提议三：评估未来退出 Style Dictionary 的可行性（长期战略）

思路：Wave 其实已经自建了 parser、resolver、transformer
和输出格式化逻辑，Style Dictionary 现在只做了最后一步的 JSON/CSS
拼接。可以在这次重构后，评估是否直接用自己的格式化代码替换 SD，把
Wave 变成一个零外部构建依赖的 CLI。

优势：

- 彻底摆脱 SD 的版本升级风险和配置心智。
- 输出格式（如 .jsonc 带注释、.css 带描述）可以完全按 UI
  设计师的需求定制。
- 执行速度可能更快（SD 的初始化开销不小）。

代价：

- 这次项目的范围会爆炸式扩大，完全不符合"中度重构"的设定。
- 需要重新处理跨平台路径、CSS 变量作用域等 SD
  已经解决好的边缘问题。

适用：这是一个备忘项，不是现在做。可以在 RESOURCE
重构完成后，单独做一次"SD 替代可行性"的 spike。
